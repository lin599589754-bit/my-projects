# 需求分析

## 1. 项目背景

本项目来源于毕业设计开题报告，题目为“本地社区生鲜配送系统及小程序设计”。

核心业务场景是面向社区用户的生鲜购买与配送服务，系统需要同时覆盖：

- 用户端微信小程序
- 商家/管理员后台
- 后端业务服务
- 数据库设计与数据管理

## 2. 重写目标

这次不沿用旧项目的实现细节，而是保留其业务范围，重写成一个结构清晰、可维护、可演示的项目。

目标如下：

1. 让系统具备完整的社区生鲜业务闭环。
2. 让数据库结构清晰，便于后续扩展。
3. 让后端接口、前端页面、数据库设计彼此对应。
4. 让项目能作为毕业设计成果展示，而不是只是一套能跑的代码。

## 3. 业务范围

项目保留的核心业务如下：

- 商品浏览
- 商品分类
- 商品详情
- 购物车
- 收货地址管理
- 下单
- 订单列表
- 订单详情
- 订单状态流转
- 管理后台商品管理
- 管理后台订单管理
- 管理后台用户管理

## 4. 角色定义

### 4.1 普通用户

普通用户通过微信小程序使用系统，主要完成：

- 查看商品
- 加入购物车
- 维护收货地址
- 提交订单
- 查看订单状态

### 4.2 管理员

管理员通过后台系统使用系统，主要完成：

- 商品上架下架
- 商品信息维护
- 订单处理
- 用户管理
- 数据查看

## 5. 功能需求

### 5.1 微信小程序端

用户端需要具备以下功能：

- 首页展示轮播、分类和推荐商品
- 分类页按分类查看商品
- 商品详情页查看价格、库存、图片、描述
- 购物车页支持增删改查和选中结算
- 地址页支持新增、编辑、删除、设置默认地址
- 确认订单页支持提交订单
- 订单页支持按状态查看订单
- 订单详情页支持查看订单明细和状态
- 个人中心页展示用户信息和订单概况

### 5.2 管理后台

后台需要具备以下功能：

- 商品管理
- 商品分类管理
- 订单管理
- 用户管理
- 首页统计展示

### 5.3 后端服务

后端需要提供以下能力：

- 用户相关接口
- 商品相关接口
- 分类相关接口
- 购物车相关接口
- 地址相关接口
- 订单相关接口
- 管理后台接口
- 基础状态流转和数据校验

## 6. 数据需求

数据库至少需要覆盖以下实体：

- 用户
- 商品
- 商品分类
- 购物车
- 地址
- 订单
- 订单明细

数据库设计原则：

- 订单主表和订单明细表分离
- 订单中保留收货信息快照
- 商品与分类分离
- 购物车支持用户和商品唯一约束
- 状态字段要有明确语义

## 7. 重写原则

重写版项目遵循以下原则：

1. 业务不乱加，先完成核心闭环。
2. 数据库结构先规范，再做前端页面。
3. 后端接口先稳定，再补管理功能。
4. 能复用的功能尽量复用业务逻辑，不做无意义重复。
5. 保留“能讲清楚”的工程结构，方便毕业设计答辩。

## 8. 初步技术方向

- 后端：Spring Boot
- 数据库：MySQL
- 管理后台：Vue
- 用户端：微信小程序
- 开发工具：IDEA、VS Code、DBeaver、微信开发者工具

## 9. 当前结论

这个项目适合重写，但不适合把业务范围扩大太多。

正确做法是：

1. 先重建数据库结构
2. 再实现后端接口
3. 再实现小程序端
4. 再实现管理后台
5. 最后补部署和说明文档
