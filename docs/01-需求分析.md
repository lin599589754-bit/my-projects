# 需求分析

## 1. 项目定位

FreshFoodSystem-v2 是一个生鲜电商系统后端重写项目。当前目标不是直接做完整商业系统，而是把原项目模型重新整理成一个结构清晰、接口可联调、适合求职展示的 Spring Boot 后端项目。

当前重点：

- 后端业务链路完整
- 数据库结构清晰
- 接口返回统一
- 参数校验和异常处理规范
- JWT 认证和用户数据隔离
- 后续可接 Web 前端

## 2. 用户角色

当前已实现的是普通用户端能力。

| 角色 | 当前状态 | 说明 |
| --- | --- | --- |
| 普通用户 | 已实现 | 浏览商品、维护地址、购物车、下单、模拟支付、取消订单、确认收货 |
| 管理员 | 暂未实现 | 后续可用于商品管理、订单发货、用户管理 |

由于管理员角色暂未实现，以下接口当前不对普通用户开放：

- 查询全部用户
- 根据 openid 查询用户
- 订单发货

## 3. 核心业务流程

当前用户端主流程：

```text
查看分类
  -> 查看商品
  -> 登录获取 JWT
  -> 新增/选择收货地址
  -> 加入购物车
  -> 创建订单
  -> 模拟支付
  -> 确认收货或取消订单
```

## 4. 功能模块

### 4.1 分类模块

用于展示启用状态的商品分类。

已实现：

- 查询启用分类
- 按排序值和 ID 升序返回

### 4.2 商品模块

用于商品浏览。

已实现：

- 查询上架商品
- 按分类查询商品
- 查询热销商品
- 查询新品商品
- 查询商品详情

暂未实现：

- 后台新增商品
- 后台修改商品
- 商品图片上传
- 商品搜索和分页

### 4.3 用户模块

用于登录和当前用户识别。

已实现：

- 登录或自动注册
- 返回 JWT
- 查询当前用户
- 根据 token 校验用户数据所有权

说明：

- 当前登录仍使用 `openid` 模拟身份来源。
- 项目已从小程序优先改为 Web 优先，因此不再强调微信 code 换 openid。
- 后续如果要接真实微信登录，可以把登录入口改成后端调用微信接口。

### 4.4 地址模块

用于维护收货地址。

已实现：

- 查询当前用户地址
- 新增地址
- 修改地址
- 设置默认地址
- 删除地址
- 新增默认地址时自动取消旧默认地址

### 4.5 购物车模块

用于下单前暂存商品。

已实现：

- 查询当前用户购物车
- 查询选中的购物车项
- 加入购物车
- 修改数量
- 修改选中状态
- 删除购物车项
- 清空购物车

业务规则：

- 加入同一商品会累加数量。
- 数量不能小于 1。
- 商品下架或库存不足时会返回业务错误。

### 4.6 订单模块

用于创建和推进订单状态。

已实现：

- 查询当前用户订单
- 按状态查询订单
- 查询订单详情
- 从购物车创建订单
- 模拟支付
- 确认收货
- 取消订单

订单状态：

| 状态值 | 含义 |
| --- | --- |
| `0` | 待付款 |
| `1` | 待发货 |
| `2` | 待收货 |
| `3` | 已完成 |
| `4` | 已取消 |

业务规则：

- 创建订单只读取选中的购物车项。
- 创建订单时会校验商品上架状态和库存。
- 创建订单后会扣减库存并清理已结算购物车项。
- 只有待付款订单可以取消。
- 只有待付款订单可以模拟支付。
- 只有待收货订单可以确认收货。
- 查询不存在的订单会返回 `404`。

## 5. 认证和权限

当前采用 JWT 认证。

公开接口：

- 健康检查
- 登录
- 分类查询
- 商品查询

受保护接口：

- 当前用户信息
- 地址
- 购物车
- 订单

权限原则：

- 前端不能随意操作其他用户的数据。
- 推荐使用 `/current` 接口，由后端从 token 中识别当前用户。
- 旧的 `/user/{userId}` 风格接口保留，但必须校验 token 中的用户 ID 是否一致。

## 6. 当前不做的范围

为控制项目范围，当前暂不实现：

- 管理员后台
- 管理员角色权限
- 真实支付
- 真实微信登录
- 秒杀级高并发库存控制
- 订单物流对接
- 部署上线

这些内容适合放在后续版本中继续扩展。

## 7. 后续优先级

建议后续顺序：

1. Web 前端联调用户端主流程。
2. 补充 Swagger/Knife4j 接口文档。
3. 做管理后台和管理员角色。
4. 优化订单号生成和库存并发控制。
5. 增加部署文档、项目截图和演示说明。
