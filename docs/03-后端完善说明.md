# 后端完善说明

本文记录 FreshFoodSystem-v2 后端在核心 CRUD 之外补充的工程化能力，以及当前仍可继续优化的点。

## 1. 当前后端链路

当前用户端链路已经完成：

```text
分类 -> 商品 -> 登录认证 -> 地址 -> 购物车 -> 订单
```

对应后端分层：

```text
浏览器 / 前端
  -> Controller 接收请求
  -> DTO 接收和校验参数
  -> Service 处理业务逻辑
  -> Repository 操作数据库
  -> MySQL 返回数据
  -> ApiResponse 统一返回前端
```

## 2. 统一响应结构

项目使用 `ApiResponse<T>` 统一返回格式。

成功：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

失败：

```json
{
  "code": 400,
  "message": "错误原因",
  "data": null
}
```

这样前端可以统一处理：

```text
code = 200    请求成功
code != 200   请求失败，展示或处理 message
```

## 3. 全局异常处理

已新增：

```text
backend/src/main/java/com/freshfood/backend/common/GlobalExceptionHandler.java
```

核心注解：

| 注解 | 含义 |
| --- | --- |
| `@RestControllerAdvice` | 声明全局控制器增强类，用于统一处理 Controller 抛出的异常 |
| `@ExceptionHandler` | 指定某个方法处理哪一种异常 |

当前异常分类：

| 异常类型 | 场景 | HTTP 状态码 | JSON code |
| --- | --- | --- | --- |
| `BusinessException` | 业务规则不满足，例如库存不足、订单状态不允许操作 | `400` | `400` |
| `NotFoundException` | 资源不存在，例如订单不存在 | `404` | `404` |
| `AccessDeniedException` | 越权访问其他用户数据 | `403` | `403` |
| `MethodArgumentNotValidException` | JSON 请求体 DTO 校验失败 | `400` | `400` |
| `BindException` | 参数绑定或表单对象校验失败 | `400` | `400` |
| `HandlerMethodValidationException` | Controller 方法参数校验失败 | `400` | `400` |
| `ConstraintViolationException` | Validation 约束校验失败 | `400` | `400` |
| `MissingServletRequestParameterException` | 缺少必要请求参数 | `400` | `400` |
| `MethodArgumentTypeMismatchException` | 参数类型错误 | `400` | `400` |
| `HttpRequestMethodNotSupportedException` | 请求方式错误 | `405` | `405` |
| `RuntimeException` | 未知运行时异常 | `500` | `500` |
| `Exception` | 其他未知异常兜底 | `500` | `500` |

关键词说明：

- `BusinessException`：业务异常，表示请求能被后端理解，但不符合业务规则。
- `NotFoundException`：资源不存在异常，表示 URL 或参数合法，但数据库中找不到对应数据。
- `RuntimeException`：运行时异常，当前作为未知内部错误兜底，不直接把内部 message 返回前端。

## 4. 参数校验

参数校验用于防止错误数据进入业务逻辑和数据库。

常用注解：

| 注解 | 含义 | 示例 |
| --- | --- | --- |
| `@NotNull` | 不能为 `null` | 用户 ID、商品 ID |
| `@NotBlank` | 字符串不能为空，也不能全是空格 | openid、收货人、手机号 |
| `@Min` | 数字不能小于指定值 | 数量至少为 1 |
| `@Max` | 数字不能大于指定值 | 状态最大为 4 |
| `@Size` | 字符串长度限制 | 备注最多 200 字 |

Controller 中配合使用：

| 注解 | 含义 |
| --- | --- |
| `@Valid` | 开启对象参数校验 |
| `@Validated` | 开启 Controller 方法级参数校验 |
| `@RequestBody` | 从 JSON 请求体读取对象 |
| `@RequestParam` | 从 URL 查询参数读取值 |
| `@PathVariable` | 从 URL 路径读取值 |

## 5. DTO 请求对象

DTO 是 Data Transfer Object，中文可以理解为“数据传输对象”。

本项目不直接让 Controller 接收 Entity，而是用 DTO 接收前端请求。

当前 DTO：

| DTO | 用途 |
| --- | --- |
| `UserLoginRequest` | 用户登录 |
| `AddressRequest` | 新增或修改地址 |
| `CurrentAddressRequest` | 当前用户新增或修改地址 |
| `CartAddRequest` | 加入购物车 |
| `OrderCreateRequest` | 创建订单 |
| `LoginResponse` | 登录成功响应 |

这样做的好处：

- Entity 专注数据库映射。
- DTO 专注接口入参和出参。
- 前端能传什么字段更清楚。
- 校验规则可以写在请求对象上。

## 6. JWT 认证收口

当前已经完成 Spring Security + JWT。

已完成：

- 登录后签发 JWT。
- token 中使用 `userId` 作为 subject。
- 地址、购物车、订单等接口需要登录。
- `/current` 接口从 token 中识别当前用户。
- 旧的 `/user/{userId}` 接口校验 token 用户 ID 和路径用户 ID 是否一致。
- 未登录返回 `401 请先登录`。
- 无权限返回 `403 无权限访问`。

当前禁止普通用户访问：

- `GET /api/users`
- `GET /api/users/openid/{openid}`
- `PUT /api/orders/{id}/ship`

原因：

- 查询全部用户和按 openid 查询用户存在用户数据枚举风险。
- 发货属于管理员能力，当前还没有管理员角色。

## 7. 订单详情 404

之前 `GET /api/orders/{id}` 找不到订单时会返回：

```json
{
  "code": 200,
  "message": "success",
  "data": null
}
```

这个语义不够规范。当前已改为：

```json
{
  "code": 404,
  "message": "订单不存在",
  "data": null
}
```

这样前端能明确区分：

- 请求成功但数据为空。
- 查询的资源不存在。

## 8. 当前测试覆盖

当前已有测试覆盖：

- Spring Boot 上下文启动
- JWT 登录返回 token
- 未登录访问受保护接口返回 401
- `/current` 接口使用 token 中的用户 ID

## 9. 当前已知但暂不处理的优化点

以下问题在高并发或大数据量场景下有意义，但当前阶段不阻塞毕设演示和 Web 前端联调。

| 问题 | 说明 | 后续方案 |
| --- | --- | --- |
| 订单号使用 `new Random()` | 同一秒高并发下理论上可能碰撞 | 数据库唯一约束 + 重试、雪花算法、ULID |
| 下单库存扣减无锁 | 高并发下可能超卖 | 悲观锁、乐观锁 version、条件更新库存 |
| `increaseSaleVolume` 存在 N+1 查询 | 一个订单多个商品时会多次查商品 | 批量查询商品或批量更新销量 |
| 缺少管理后台 | 发货、商品管理、用户管理暂未开放 | 增加管理员角色和后台接口 |
| 缺少接口文档页面 | 当前是 Markdown 文档 | 后续可接 Swagger/Knife4j |

## 10. 当前结论

当前后端已经具备用户端主流程联调能力，也具备较基础的工程化质量：

```text
统一返回 -> 参数校验 -> 全局异常 -> JWT认证 -> 所有权校验 -> 接口测试
```

下一步建议优先做 Web 前端联调，通过真实页面验证接口是否顺手、字段是否够用、流程是否闭环。
