# 后端完善说明

本文记录 FreshFoodSystem-v2 后端在核心业务接口完成后需要补充的工程化内容。

当前用户端核心链路已经基本完成：

```text
分类 -> 商品 -> 用户 -> 地址 -> 购物车 -> 订单
```

后续为了让后端更稳定、更适合前端对接，需要优先补充：

1. 全局异常处理
2. 参数校验

## 1. 为什么要做全局异常处理

现在后端接口成功时统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

但如果业务代码中抛出异常，例如：

```java
throw new RuntimeException("库存不足");
```

如果没有统一处理，Spring 可能返回默认错误结构，格式和正常响应不一致，也可能暴露较多内部错误信息。

全局异常处理的目标是：让错误响应也保持统一格式。

预期错误返回：

```json
{
  "code": 500,
  "message": "库存不足",
  "data": null
}
```

这样前端可以统一判断：

```text
code = 200    请求成功
code != 200   请求失败，展示 message
```

## 2. 全局异常处理计划

计划新增文件：

```text
backend/src/main/java/com/freshfood/backend/common/GlobalExceptionHandler.java
```

主要使用注解：

| 注解 | 含义 |
| --- | --- |
| `@RestControllerAdvice` | 声明全局控制器增强类，可以统一处理 Controller 抛出的异常 |
| `@ExceptionHandler` | 指定某个方法处理哪一种异常 |

示例结构：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(RuntimeException.class)
    public ApiResponse<Void> handleRuntimeException(RuntimeException e) {
        return ApiResponse.fail(500, e.getMessage());
    }
}
```

说明：

- `RuntimeException`：运行时异常，当前 Service 中很多业务错误都是通过它抛出的。
- `ApiResponse.fail(...)`：复用当前项目已有的统一失败返回方法。
- `Void`：表示失败响应中没有业务数据，`data` 为 `null`。

## 3. 为什么要做参数校验

参数校验用于防止错误数据进入业务逻辑和数据库。

例如新增地址时，如果前端传入：

```json
{
  "receiverName": "",
  "receiverPhone": "",
  "detailAddress": ""
}
```

没有参数校验时，问题可能到数据库保存阶段才暴露，错误提示不清楚。

加入参数校验后，后端可以提前返回：

```json
{
  "code": 400,
  "message": "收货人不能为空",
  "data": null
}
```

参数校验的作用：

- 提前拦截错误请求。
- 减少 Service 中重复的 `if` 判断。
- 保护数据库数据质量。
- 给前端更清晰的错误提示。

## 4. 参数校验计划

当前项目已经引入了 Validation 依赖，可以直接使用 Jakarta Validation 注解。

常用注解：

| 注解 | 含义 | 示例 |
| --- | --- | --- |
| `@NotNull` | 不能为 `null` | 用户 ID、商品 ID |
| `@NotBlank` | 字符串不能为空，也不能全是空格 | 收货人、手机号、详细地址 |
| `@Min` | 数字不能小于指定值 | 数量至少为 1 |
| `@Max` | 数字不能大于指定值 | 状态值最大为 4 |
| `@Size` | 字符串或集合长度限制 | 备注最多 200 字 |

Controller 中需要配合使用：

| 注解 | 含义 |
| --- | --- |
| `@Valid` | 校验 `@RequestBody` 接收的对象 |
| `@Validated` | 开启 Controller 方法参数校验 |

## 5. 参数校验的推荐落点

为了不把 Entity 写得太臃肿，后续更推荐新增请求对象，也就是 DTO。

DTO 的意思是 Data Transfer Object，中文可以理解为“数据传输对象”。

例如新增地址时，不直接让 Controller 接收 `Address`，而是接收：

```text
AddressRequest
```

计划目录：

```text
backend/src/main/java/com/freshfood/backend/dto/
```

示例：

```java
public class AddressRequest {

    @NotNull(message = "用户ID不能为空")
    private Long userId;

    @NotBlank(message = "收货人不能为空")
    private String receiverName;

    @NotBlank(message = "收货电话不能为空")
    private String receiverPhone;
}
```

这样做的好处：

- Entity 专注对应数据库表。
- DTO 专注接收前端参数。
- 哪些字段允许前端传，哪些字段由后端生成，会更清楚。

## 6. 本项目的实施顺序

建议按以下顺序实现：

```text
1. 新增 GlobalExceptionHandler
2. 处理 RuntimeException
3. 处理参数校验异常
4. 给简单参数接口加校验
5. 给地址、购物车、订单等写 DTO
6. 测试错误返回格式
```

第一步可以先只做全局异常处理，因为它改动小、收益明显。

## 7. 预期错误码

项目当前先使用简单错误码：

| code | 含义 |
| --- | --- |
| `200` | 成功 |
| `400` | 请求参数错误 |
| `404` | 数据不存在 |
| `500` | 业务错误或服务器错误 |

后续如果项目规模变大，可以再细分错误码。

## 8. 测试方式

全局异常处理完成后，可以用以下方式测试。

库存不足：

```http
POST /api/carts?userId=1&productId=1&quantity=999999
```

预期返回：

```json
{
  "code": 500,
  "message": "库存不足",
  "data": null
}
```

订单没有选中商品：

```http
POST /api/orders?userId=999&addressId=1
```

预期返回：

```json
{
  "code": 500,
  "message": "购物车中没有选中的商品",
  "data": null
}
```

参数校验完成后，还需要测试：

```http
POST /api/addresses
Content-Type: application/json
```

请求体：

```json
{
  "userId": 1,
  "receiverName": "",
  "receiverPhone": "",
  "province": "",
  "city": "",
  "district": "",
  "detailAddress": ""
}
```

预期返回：

```json
{
  "code": 400,
  "message": "收货人不能为空",
  "data": null
}
```

## 9. 当前结论

全局异常处理和参数校验不是新增业务功能，但它们能显著提升项目质量。

它们解决的问题分别是：

```text
全局异常处理：错误发生后，如何统一返回给前端。
参数校验：请求进入业务逻辑前，如何判断数据是否合法。
```

建议下一步先实现全局异常处理，再逐步给重点接口补参数校验。
