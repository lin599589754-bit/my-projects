# 后端完善说明

本文记录 FreshFoodSystem-v2 后端在核心业务接口完成后需要补充的工程化内容。

当前用户端核心链路已经基本完成：

```text
分类 -> 商品 -> 用户 -> 地址 -> 购物车 -> 订单
```

后续为了让后端更稳定、更适合前端对接，需要优先补充：

1. 全局异常处理
2. 参数校验

## 1. 为什么要做全局异常处理

现在后端接口成功时统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

但如果业务代码中抛出异常，例如：

```java
throw new RuntimeException("库存不足");
```

如果没有统一处理，Spring 可能返回默认错误结构，格式和正常响应不一致，也可能暴露较多内部错误信息。

全局异常处理的目标是：让错误响应也保持统一格式，并且让 HTTP 状态码和响应体中的 `code` 保持一致。

预期错误返回示例：

```json
{
  "code": 500,
  "message": "库存不足",
  "data": null
}
```

这样前端可以统一判断：

```text
code = 200    请求成功
code != 200   请求失败，展示 message
```

同时，HTTP 状态码也能表达请求结果：

```text
HTTP 200   请求成功
HTTP 400   请求参数错误
HTTP 405   请求方式错误
HTTP 500   业务异常或服务器异常
```

## 2. 全局异常处理实现

已新增文件：

```text
backend/src/main/java/com/freshfood/backend/common/GlobalExceptionHandler.java
```

主要使用注解：

| 注解 | 含义 |
| --- | --- |
| `@RestControllerAdvice` | 声明全局控制器增强类，可以统一处理 Controller 抛出的异常 |
| `@ExceptionHandler` | 指定某个方法处理哪一种异常 |

主要使用类：

| 类 | 含义 |
| --- | --- |
| `ResponseEntity` | Spring 提供的响应对象，可以同时设置 HTTP 状态码和响应体 |
| `HttpStatus` | Spring 提供的 HTTP 状态码枚举，例如 `BAD_REQUEST`、`METHOD_NOT_ALLOWED` |

示例结构：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(RuntimeException.class)
    public ResponseEntity<ApiResponse<Void>> handleRuntimeException(RuntimeException e) {
        return ResponseEntity
                .status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.fail(500, e.getMessage()));
    }
}
```

说明：

- `RuntimeException`：运行时异常，当前 Service 中很多业务错误都是通过它抛出的。
- `ApiResponse.fail(...)`：复用当前项目已有的统一失败返回方法。
- `Void`：表示失败响应中没有业务数据，`data` 为 `null`。
- `ResponseEntity`：用于让 HTTP 状态码和 JSON 中的 `code` 同步。

当前已处理的异常类型：

| 异常类型 | 场景 | HTTP 状态码 | JSON code |
| --- | --- | --- | --- |
| `RuntimeException` | 业务异常，例如库存不足、购物车没有选中商品 | `500` | `500` |
| `MissingServletRequestParameterException` | 缺少必要请求参数 | `400` | `400` |
| `MethodArgumentTypeMismatchException` | 参数类型错误，例如 `userId=abc` | `400` | `400` |
| `MethodArgumentNotValidException` | `@RequestBody` DTO 校验失败 | `400` | `400` |
| `BindException` | `@ModelAttribute` DTO 校验失败 | `400` | `400` |
| `HandlerMethodValidationException` | Controller 方法参数校验失败 | `400` | `400` |
| `ConstraintViolationException` | Validation 约束校验失败 | `400` | `400` |
| `HttpRequestMethodNotSupportedException` | 请求方式错误，例如应使用 POST 却使用 GET | `405` | `405` |
| `Exception` | 未知异常兜底处理 | `500` | `500` |

## 3. 为什么要做参数校验

参数校验用于防止错误数据进入业务逻辑和数据库。

例如新增地址时，如果前端传入：

```json
{
  "receiverName": "",
  "receiverPhone": "",
  "detailAddress": ""
}
```

没有参数校验时，问题可能到数据库保存阶段才暴露，错误提示不清楚。

加入参数校验后，后端可以提前返回：

```json
{
  "code": 400,
  "message": "收货人不能为空",
  "data": null
}
```

参数校验的作用：

- 提前拦截错误请求。
- 减少 Service 中重复的 `if` 判断。
- 保护数据库数据质量。
- 给前端更清晰的错误提示。

## 4. 参数校验实现

当前项目已经引入了 Validation 依赖，可以直接使用 Jakarta Validation 注解。

常用注解：

| 注解 | 含义 | 示例 |
| --- | --- | --- |
| `@NotNull` | 不能为 `null` | 用户 ID、商品 ID |
| `@NotBlank` | 字符串不能为空，也不能全是空格 | 收货人、手机号、详细地址 |
| `@Min` | 数字不能小于指定值 | 数量至少为 1 |
| `@Max` | 数字不能大于指定值 | 状态值最大为 4 |
| `@Size` | 字符串或集合长度限制 | 备注最多 200 字 |

Controller 中需要配合使用：

| 注解 | 含义 |
| --- | --- |
| `@Valid` | 校验 `@RequestBody` 接收的对象 |
| `@Validated` | 开启 Controller 方法参数校验 |

当前已完成的校验范围：

| 模块 | 校验位置 | 说明 |
| --- | --- | --- |
| 地址 | `AddressRequest` | 新增、修改地址时校验用户 ID、收货人、手机号、省市区、详细地址、默认状态 |
| 购物车 | `CartAddRequest` 和 Controller 参数 | 加入购物车时校验用户 ID、商品 ID、数量；更新数量和选中状态时校验范围 |
| 订单 | `OrderCreateRequest` 和 Controller 参数 | 创建订单时校验用户 ID、地址 ID、备注长度；订单状态和订单 ID 校验范围 |
| 用户 | `UserLoginRequest` 和 Controller 参数 | 登录时校验 `openid`，查询时校验用户 ID 和 `openid` 长度 |
| 商品 | Controller 参数 | 查询分类商品和商品详情时校验 ID |

## 5. 参数校验的推荐落点

为了不把 Entity 写得太臃肿，本项目已经新增请求对象，也就是 DTO。

DTO 的意思是 Data Transfer Object，中文可以理解为“数据传输对象”。

例如新增地址时，不直接让 Controller 接收 `Address`，而是接收：

```text
AddressRequest
```

目录：

```text
backend/src/main/java/com/freshfood/backend/dto/
```

示例：

```java
public class AddressRequest {

    @NotNull(message = "用户ID不能为空")
    private Long userId;

    @NotBlank(message = "收货人不能为空")
    private String receiverName;

    @NotBlank(message = "收货电话不能为空")
    private String receiverPhone;
}
```

这样做的好处：

- Entity 专注对应数据库表。
- DTO 专注接收前端参数。
- 哪些字段允许前端传，哪些字段由后端生成，会更清楚。

当前已新增 DTO：

```text
AddressRequest
CartAddRequest
OrderCreateRequest
UserLoginRequest
```

## 9. 当前结论

全局异常处理和参数校验不是新增业务功能，但它们能显著提升项目质量。

它们解决的问题分别是：

```text
全局异常处理：错误发生后，如何统一返回给前端。
参数校验：请求进入业务逻辑前，如何判断数据是否合法。
```

当前全局异常处理、参数校验、基础 DTO 和 Spring Security + JWT 登录认证已经完成。

下一步建议开始 Web 前端对接前的接口梳理，确认用户端 Web 和管理后台需要的返回字段是否足够。
